## Codebase Patterns
- SimpleTable is now a public top-level class at `api/src/main/java/org/opensearch/sql/api/SimpleTable.java` — use it directly, no testFixtures dependency needed
- Spotless uses Google Java Format 1.32.0 with Apache 2.0 license header — run `./gradlew :api:spotlessApply` before build
- JaCoCo enforces 90% coverage on api module — new main source code needs tests or must be covered by existing tests
- Same-package classes in testFixtures can reference main source set classes without explicit imports (both are `org.opensearch.sql.api`)
- Build command: `./gradlew :api:build` runs compile + test + jacoco + spotless check
- Jackson (ObjectMapper, databind) is available transitively in cli via cli → api → ppl → core — no explicit dependency needed
- Jackson deserializes JSON integers as Integer or Long — cast Long to int via .intValue() when mapping to SqlTypeName.INTEGER
- hr.json dataset: employees (id, name, age, department, salary) × 4 rows; departments (id, name, budget) × 3 rows
- Query execution flow: UnifiedQueryContext.builder().language(QueryType).catalog(name, schema).defaultNamespace(name).build() → new UnifiedQueryPlanner(ctx).plan(query) → new UnifiedQueryCompiler(ctx).compile(plan) → stmt.executeQuery()
- SyntaxCheckException is at org.opensearch.sql.common.antlr.SyntaxCheckException — catch it for user-friendly syntax error messages
- UnifiedQueryContext is AutoCloseable — always close before rebuilding or on exit
- For testing CLI classes that need real query execution, use SampleDataLoader.loadFromClasspath("data/hr.json") to get tables, then construct the context/planner/compiler directly
- Package-private methods (loop, dispatch, prompt) enable unit testing of REPL logic without a real JLine terminal
- Gradle `application` plugin `run` task defaults workingDir to subproject dir — use `run { workingDir = rootProject.projectDir }` for root-relative paths

  - US-008: Added QueryHighlighter with keyword/string/number/meta-command syntax highlighting
  - US-009: Created ecommerce.json dataset, verified all 4 end-to-end acceptance criteria
- **Remaining**: None — all stories complete and passing
- **Blockers**: None. The loop used 31 iterations for 9 stories (~3.4 iterations per story average), suggesting some stories required retry cycles for spotless/build fixes but nothing was fundamentally blocked.

### PoC Direction Check
- **Approach validation**: The approach is fully viable. Calling UnifiedQueryPlanner/UnifiedQueryCompiler directly from a JLine3 REPL works cleanly. The plan→compile→executeQuery→format pipeline is straightforward and the Calcite-backed engine handles both SQL and PPL without issues. SimpleTable as a public API class is a clean abstraction for in-memory data.
- **Surprises**: 
  - Easier than expected: Jackson being transitively available via Calcite eliminated a dependency concern. Same-package resolution between testFixtures and main source set meant US-001 required zero test modifications.
  - Harder than expected: Gradle `application` plugin defaults `workingDir` to the subproject directory, not root — caused relative path issues in US-009 that needed a build.gradle fix. Spotless formatting likely caused retry cycles across multiple stories.
- **Alternative approaches**: None surfaced. The JLine3 + PicoCLI + direct API call approach proved clean. No indication that an alternative architecture would be better for this PoC scope.
- **Risk areas**: 
  - JaCoCo coverage threshold is 0.0 for cli module — if this moves beyond PoC, coverage enforcement should be raised.
  - The highlighter and completer maintain their own keyword lists separate from the actual parser grammars — these will drift as SQL/PPL syntax evolves.
  - No integration test for the interactive REPL path (only unit tests with mocked LineReader) — terminal-dependent behavior could have edge cases.
- **Open questions**: 
  - Should SimpleTable remain in the api module long-term, or should it move to a shared module if other consumers emerge?
  - Should the CLI be distributed as a fat JAR / native image for easier end-user consumption?
  - Should keyword lists be extracted from the ANTLR grammars rather than hardcoded?

### Recommended Next Steps
- Raise JaCoCo coverage threshold for cli module from 0.0 to a reasonable level (e.g., 80%)
- Consider extracting SQL/PPL keyword lists from ANTLR grammar files to keep completer/highlighter in sync with parser
- If shipping beyond PoC: add shadow/fat JAR task or GraalVM native-image build for standalone distribution
- Add a brief README.md to cli/ with usage examples and screenshots
- Consider adding multi-line query support (semicolon-terminated) to the REPL
