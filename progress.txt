### Array/MV Function Patterns
- Use SqlLibraryOperators.FUNC for array functions (ARRAY_DISTINCT, SPLIT, SORT_ARRAY, ARRAY_COMPACT, ARRAY, ARRAY_CONCAT, ARRAYS_ZIP, ARRAY_SLICE) — NOT call("FUNC") which creates SqlBasicFunction with wrong return type
- ARRAY() with string literals: cast to VARCHAR to prevent CHAR padding
- UDFs must be registered in BOTH lowercase and uppercase (SqlPrettyWriter uppercases all function names)
- RestUnifiedSQLQueryAction needs explicit java.sql.Types.ARRAY handling for JDBC response
- ARRAY_SLICE requires INTEGER args — arithmetic expressions produce NUMERIC (engine limitation)
- Lambda functions (forall, exists, filter, transform, reduce, mvmap) have no SQL equivalent — engine-blocked

### Bin Command Patterns
- All bin types use subquery approach: SELECT *, binExpr AS _bin_field FROM (subquery)
- binReplacements map tracks field→expression for downstream pipe resolution
- SpanBin: numeric → FLOOR(field/span)*span CONCAT range; time → floorToUnit() or epoch bucketing
- CountBin: POWER(10, CEIL(LOG10(range/bins))) with MIN/MAX OVER() window functions
- MinSpanBin: magnitude-based width with CASE WHEN >= minspan constraint
- RangeBin: magnitude-based width with optional start/end range expansion
- DefaultBin: automatic magnitude-based POWER(10, FLOOR(LOG10(range)))
- Time-based bin results must be CAST to VARCHAR (tests expect string type)
- floorToUnit() helper: SqlStdOperatorTable.FLOOR + SqlIntervalQualifier for FLOOR(field TO HOUR) syntax

### Chart/Timechart Patterns
- Chart and timechart share the same Chart AST node — single visitChart handles both
- Chart = aggregation + optional column split: SELECT rowSplit, colSplit, AGG() GROUP BY rowSplit, colSplit
- Chart.transformPerFunction() rewrites per_* functions at AST level before converter sees them
- Null row splits always filtered (WHERE rowSplit IS NOT NULL)
- Column splits cast to VARCHAR for nullstr/otherstr compatibility
- usenull=true (default): COALESCE(CAST(colSplit AS VARCHAR), nullstr)
- Chart limit/useother requires ROW_NUMBER ranking + secondary aggregation (not yet implemented)

## Codebase Patterns

### PPL AST Structure

### Schema Resolution (CRITICAL)
- `SchemaPlus.getTable()` does NOT trigger lazy `HashMap.get()` in `AbstractSchema.getTableMap()` — it goes through `CalciteSchema.tables()` Lookup which bypasses the override
- Must use `CalciteSchema.from(schemaPlus).schema.getTable(tableName)` to access the underlying Schema directly
- This affects ALL schema-dependent operations: eval override, fields-, rename wildcard, fillnull all-fields
- Unit tests work because they use simple in-memory schemas; integration tests use lazy-loading AbstractSchema

### PPL AST Structure
- PPL parser always wraps plans in Project(AllFields) — treat as passthrough
- PPL count() AST uses Literal(1) not AllFields — check for both
- Pipe commands chain as nested AST: Sort(Aggregation(Filter(Relation))) — process inside-out
- Join aliases (left=l right=r) are in the AST and must appear in SQL output
- SpanBin has field, value, unit; uppercase M = MONTH, lowercase m = MINUTE

### SqlNode Converter Patterns
- Each PPL pipe command wraps the previous result as a subquery (inside-out construction)
- Use SqlNodeDSL static factory methods: select(), from(), where(), groupBy(), orderBy(), etc.
- Use SqlBasicFunction.create() for unquoted function names — new SqlFunction() quotes them
- Deferred ORDER BY: use pendingOrderBy/pendingFetch fields to carry ORDER BY across pipe stages
- CAST to INTEGER for datetime EXTRACT functions — Calcite EXTRACT returns BIGINT
- PPLTypeCoercion handles implicit type coercion during validation — converter doesn't emit CASTs
- DynamicPPLToSqlNodeConverter uses SchemaPlus only for column enumeration (fields -, rename wildcards, fillnull all-fields)

### Function Mapping
- PPL functions map to SQL via FUNC_MAP (static map in converter)
- PPL-specific aggregates: first→PPL_FIRST, last→PPL_LAST, take→TAKE, percentile_approx→UDF
- Arithmetic: add→+, subtract→-, multiply→*, divide→/, modulus→MOD
- Now-like functions: generate UTC string literals at transpile time (constant within query execution)
- replace()→REGEXP_REPLACE (PPL replace is regex-aware), regexp_match→REGEXP_CONTAINS
- Back-reference conversion in replace: \\N→$N (PPL Java-style → Calcite style)
- SQL datetime keywords (CURRENT_TIMESTAMP, CURRENT_DATE, etc.) must NOT have parentheses

### Aggregation Patterns
- DATE_TRUNC argument order: DATE_TRUNC(field, UNIT) — Calcite BigQuery dialect, NOT PostgreSQL
- percentile_approx: UDF call, not PERCENTILE_CONT WITHIN GROUP
- bucket_nullable=false: add WHERE clause filtering null group-by keys
- TAKE: TAKE(field, N) UDF with count argument
- COUNT(*) not COUNT() — Calcite rejects zero-argument COUNT

### Subquery Wrapping Rules
- Eval aliases need subquery wrapping so downstream pipes can reference them
- When pipe stages conflict (WHERE after GROUP BY), auto-wrap as subquery
- ORDER BY lost in subquery wrapping — save/restore orderBy+limit around wrapAsSubquery
- head-then-sort: if limit already set from prior head, wrap as subquery before applying sort

### SELECT * EXCEPT/REPLACE
- fields- exclusion: SELECT * EXCEPT(col1, col2)
- eval column overriding: SELECT * REPLACE(expr AS col)
- bin field replacement: SELECT * REPLACE(bin_expr AS field) when alias matches field
- fillnull all-fields: SELECT * REPLACE(COALESCE(col, val) AS col, ...)
- fillnull per-field: SELECT * REPLACE(COALESCE(field, value) AS field) via SqlStarExceptReplace
- replace command: SELECT * REPLACE(REPLACE(field, pattern, replacement) AS field) via SqlStarExceptReplace
- rename: SELECT * EXCEPT(old_col), old_col AS new_name — or explicit column list when resolveColumns available
- rename: subquery wrap with alias mapping
- IMPORTANT: SqlStarExceptReplace with REPLACE works without resolveColumns() — the StarExceptReplaceRewriter expands * during validation when schema IS available
- IMPORTANT: For narrowed pipes (after fields command), * REPLACE doesn't work — use extractPipeColumns() to get actual column list from pipe's SELECT list
- resolveColumns() returns metadata fields (_id, _index, _score, etc.) — always filter with !name.startsWith("_")

### Test Harness
- V4 mode throws exceptions client-side (SyntaxCheckException, IllegalArgumentException) not server-side (ResponseException) — error-handling tests systematically fail
- Escaped double-quotes: V4 path needs query.replace("\"", "\"") preprocessing
- ORDER BY column stripping: Calcite adds ORDER BY columns to response even when not in SELECT
- JSON escaping: multi-line SQL from toSqlString() breaks JSON response serialization
- CalcitePPLPluginIT is flaky — ignore ±1 fluctuations

### Join/Subquery Patterns
- visitJoin sets `pipe = joinNode` (raw SqlJoin, NOT wrapped in SELECT *) to preserve alias visibility
- `skipNextWrap` flag: set after visitJoin, consumed ONLY by visitProject. Other visitors always wrap.
- `consumeSkipWrap()`: returns pipe directly without wrapping, used by visitProject when skipNextWrap=true
- Shared `AtomicInteger aliasCounter`: passed to sub-converters via convertSubPlan() to prevent alias collision (_t1 reuse)
- Field-list joins: buildFieldListCondition uses effectiveLeftAlias/effectiveRightAlias as fallback when explicit aliases are null
- extractAlias(SqlNode): extracts alias from AS expressions or bare identifiers
- Duplicate column aliasing: visitProject pre-scans leaf names, only aliases second occurrence of duplicates
- EXCEPT/REPLACE with qualified names DOES NOT WORK — parser can't handle dots in EXCEPT/REPLACE clauses
- convert() wraps raw SqlJoin in SELECT * if it's the final result
- Lookup APPEND: DynamicPPLToSqlNodeConverter checks schema — COALESCE for columns in both tables, direct _r.col for lookup-only columns

### Engine Capabilities (do NOT modify engine)
- SqlLibrary.MYSQL/BIG_QUERY/SPARK/POSTGRESQL all registered — hundreds of functions available

### Object/Dotted Field Names
- OpenSearch object fields (city.name, employer.name) are single columns with dots — NOT schema-qualified references
- visitQualifiedName: if first part is in knownAliases → compound SqlIdentifier (table.column). Otherwise → single-part quoted identifier ("city.name")
- knownAliases tracks: join aliases (effectiveLeftAlias/effectiveRightAlias), subquery aliases (_tN), lookup aliases (_l, _r), table name

### Engine Capabilities (do NOT modify engine)
- OpenSearch date/time fields → standard SQL TIMESTAMP/DATE/TIME (no UDT)
- Metadata fields excluded from schema — SELECT * returns only user data columns
- SELECT * EXCEPT/REPLACE supported at SqlNode layer
- Custom PPL UDFs: pplFirst, pplLast, matchPhrase, percentile_approx, take, listAgg, valuesAgg

### toSqlString() Backslash Escaping (CRITICAL)
- CalciteSqlDialect.DEFAULT.toSqlString() doubles ALL backslashes in string literals
- The SQL endpoint's parser (SqlParserUtil.parseString) does NOT un-escape doubled backslashes
- Fix: transpileV4() applies sql.replace("\\\\", "\\") after toSqlString() to un-double
- This affects ALL string literals: regex patterns, escape sequences, file paths, etc.

## Human Intervention — PPLToSqlNodeConverter Refactor (2026-03-10)

The string-based PPLToSqlTranspiler (58.5%, 1324/2264) was replaced with a SqlNode-based approach:

### What was created
- **PPLToSqlNodeConverter** — extends `AbstractNodeVisitor<SqlNode, Void>`, base converter for static commands (where, sort, head, eval, stats, dedup, join, lookup, subquery, append, top, rare, trendline, streamstats)
- **DynamicPPLToSqlNodeConverter** — subclass with SchemaPlus dependency for schema-dependent commands (fields include/exclude, eval column override, rename wildcard, fillnull all-fields, replace, bin, appendcol, graphlookup TVF)
- **SqlNodeDSL** — thin utility layer of static factory methods wrapping Calcite SqlNode constructors for declarative query building
- **PPLTypeCoercion** — extends TypeCoercionImpl with MYSQL_5 conformance, adds DATE+TIME→TIMESTAMP merge rule. Fires during Calcite validation to inject explicit CASTs.
- Wired into PPLIntegTestCase V4 routing path

### Key design decisions
- Converter produces SqlNode (not RelNode) — feeds into Calcite's validate → rel pipeline
- Lighter dependency than V3's CalcitePlanContext (no RelBuilder, no RexBuilder)
- Converter does NOT track types or emit CASTs — PPLTypeCoercion handles coercion during validation
- Two-tier design: base (no schema) vs dynamic (schema-aware) for testability

### What was carried over from string transpiler
- Deferred ORDER BY pattern (pendingOrderBy/pendingFetch)
- Datetime function handling (ensureTimestamp, now-like UTC literals)
- PPL-specific aggregate mappings (PPL_FIRST, PPL_LAST, TAKE, percentile_approx)
- Function quoting fix (SqlBasicFunction.create for unquoted names)
- JSON escaping for multi-line SQL

### Current state
- Rate: 41.3% (934/2264), 414 regressions vs 58.5% string transpiler baseline

## Human Intervention — Root Cause Analysis (2026-03-11)

### 1. V4 Error Handling Path (fix in next iteration)
V4 transpilation errors throw client-side exceptions (SyntaxCheckException, IllegalArgumentException) instead of ResponseException. This is because `transpileV4()` runs before the HTTP call — if it fails, the server is never reached. ~49 test references to ResponseException are affected across many IT classes.

**Fix:** In `PPLIntegTestCase.executeQuery()` and `executeQueryToString()`, wrap `transpileV4()` in try-catch that converts client-side exceptions into ResponseException. This is assigned to V4-005 as a prerequisite task.

### 2. UDF Registration Clarification
The UDFs registered in UnifiedQueryContext (PPL_FIRST, PPL_LAST, TAKE, percentile_approx, MATCH_PHRASE) were ALL pre-existing before the PoC — they represent PPL-specific aggregates with no standard SQL equivalent. The converter correctly maps to them. No new UDFs were registered by the Ralph loop or the converter refactor. Do NOT register new UDFs without approval — represent everything in SQL first.

### 3. timestampStringToEpochMillis Bug (engine-level, out of scope)
`CalciteNoPushdownIndexScan.timestampStringToEpochMillis()` throws NumberFormatException on custom date formats (epoch_millis, yyyy-MM-dd without time). This blocks ~9 tests. This is an engine scan bug, not a converter issue — will be fixed separately outside the Ralph loop.

### 4. Search/Relevance Functions May Be Blocked
CalciteSearchCommandIT (46 failures) and CalciteRelevanceFunctionIT (14 failures) may require UDF/TVF registration for full-text search functions (match, query_string). If V4-005 is blocked by this, skip to V4-006. These are deferred to V4-026/V4-028 at the tail of the PRD.

## Human Intervention — Engine Fixes + Converter Guidance (2026-03-12)

### Engine Fix 1: Nanosecond timestamp parsing (DONE, ~45 tests unblocked)
- **Root cause**: `CalciteNoPushdownIndexScan.timestampStringToEpochMillis()` line 254 used `DateTimeFormatter.ofPattern("yyyy-MM-dd['T'][ ]HH:mm:ss[.SSS][.SSSSSS]")` which only matches exactly 3 or 6 fractional digits. OpenSearch `date_nanos` fields return 9-digit nanosecond timestamps (e.g., `2024-01-15 10:30:25.678901234`) which matched neither optional group, causing all fallback parsers to fail and ultimately throwing `NumberFormatException` from `Long.parseLong()`.
- **Fix**: Replaced hardcoded pattern with `DateTimeFormatterBuilder.appendFraction(ChronoField.NANO_OF_SECOND, 0, 9, true)` which handles 0-9 fractional digits. The codebase already had this exact pattern in `DateTimeFormatters.DATE_TIME_FORMATTER_VARIABLE_NANOS`.
- **File changed**: `opensearch/src/main/java/org/opensearch/sql/opensearch/storage/scan/CalciteNoPushdownIndexScan.java`
- **Impact**: Unblocks ~45 tests across CalciteSearchCommandIT (38), CalcitePPLAggregationIT (~7), and others that query `date_nanos` fields through the no-pushdown scan path.

### Engine Fix 2: MULTISET→ARRAY type mapping (DONE, ~20 tests unblocked)
- **Root cause**: `OpenSearchTypeFactory.convertSqlTypeNameToExprType()` line 226 grouped `MULTISET` with `CHAR, VARCHAR` → `STRING`. Some Calcite operators return `MULTISET` instead of `ARRAY` for collection types, causing array functions (ARRAY(), SPLIT(), ARRAY_DISTINCT(), ARRAY_JOIN()) to report `type=string` instead of `type=array` in JSON response metadata.
- **Fix**: Separated `MULTISET` from the `CHAR, VARCHAR` case and mapped it to `ARRAY` instead of `STRING`.
- **File changed**: `core/src/main/java/org/opensearch/sql/calcite/utils/OpenSearchTypeFactory.java`
- **Impact**: Unblocks ~20 CalciteArrayFunctionIT tests that assert `verifySchema(actual, schema("result", "array"))`.

### Engine Note: Error message format mismatches (~30 tests, NOT fixable)
- **Root cause**: V4 transpilation errors produce fundamentally different exception messages than the PPL engine. `createSyntheticResponseException()` wraps raw Java exceptions with `reason: "V4 transpilation error"`, but tests expect PPL-domain-specific messages like `"date:09:07:42 in unsupported format"` or `"function expected {[BOOLEAN,BOOLEAN],...}"`. These are PPL engine error handling tests, not transpilation correctness tests.
- **Recommendation**: Exclude from 85% target or skip when V4 is enabled. Not worth fixing — the errors are architecturally different.
