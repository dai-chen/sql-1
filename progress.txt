  - US-001: Directory structure and shared schema (test_tables.yaml with logs/employees tables)
  - US-002–US-008: Substrait YAML specs and test DSL files for all 93 PPL functions across 12 categories (IP, JSON, datetime, math, condition, aggregation, collection, parse, type_conv, crypto, rex, binning, relevance)

  - US-011–US-014: All evaluable UDFs implemented — 8 IP, 9 JSON, 33 datetime, 47 remaining (math/condition/collection/parse/type_conv/crypto/rex/binning/aggregation) = 97 total scalar UDFs
  - US-015: Schema loader + 16 E2E tests covering 9 UDF categories
- **Remaining**: None. All stories pass. 40 total tests (20 unit + 24 spec integration + 16 E2E, some counted together as 40).
- **Blockers**: None. All builds and tests green. Relevance functions (7) intentionally skipped as pushdown-only per PRD. geoip basic tests excluded (requires external MaxMind DB). ntile excluded from binning spec tests (window function, not scalar).

### PoC Direction Check
- **Approach validation**: The approach is fully viable. Defining functions as Substrait YAML specs with a test DSL, then implementing as DataFusion ScalarUDFs, works end-to-end. The spec-driven test harness catches regressions and enforces correctness against the spec. The columnar Arrow API pattern scales well across all function categories.
- **Surprises**:
  - Easier than expected: The Substrait YAML + test DSL convention was straightforward to establish and replicate across 12 categories. Once the IP/JSON patterns were set, remaining categories followed quickly.
  - Harder than expected: DataFusion type coercion — bare integer literals infer as Int64, requiring explicit CAST(val AS INT) for Int32 UDF signatures. Arrow timestamp display uses 'T' separator requiring normalization. MySQL-compatible datetime semantics (TO_DAYS algorithm, weekday numbering, period arithmetic) needed careful implementation.
  - Assumptions invalidated: None fundamental. The PRD listed "8 remaining categories" but there were actually 10 (math, condition, aggregation, collection, parse, type_conv, crypto, rex, binning, relevance). This was absorbed without issue.
- **Alternative approaches**: None needed. The current spec-first approach with Rust UDFs is clean and extensible. One consideration: aggregation functions are tested with scalar semantics (single-value input) which is a simplification — real aggregate testing would need multi-row input via the E2E harness.
- **Risk areas**:
  - Aggregation functions (percentile, stddev_*, var_*, take) are implemented as scalar UDFs but are semantically aggregate — integrating with DataFusion's aggregate framework would require rework.
  - geoip requires an external MaxMind database at runtime — no way to test without it.
  - Relevance functions (7) are pushdown-only stubs — if the engine ever needs to evaluate them locally, full implementations would be needed.
  - The test harness doesn't yet support multi-row assertions or table-level test cases (only scalar function calls).
- **Open questions**: None blocking. The PoC is complete as scoped.

### Recommended Next Steps
- The PRD is 100% complete. If resuming, consider:
  1. Integrate the Rust crate with the OpenSearch SQL/PPL engine (transpile PPL → SQL → DataFusion execution)
  2. Add proper aggregate function support (DataFusion AggregateUDFImpl trait) for percentile/stddev/var/take
  3. Add multi-row E2E tests for aggregate functions
  4. Set up CI pipeline for the Rust crate (cargo build + cargo test)
  5. Add geoip implementation with bundled test database
